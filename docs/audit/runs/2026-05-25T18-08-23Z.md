Exercised the profile menu dropdown reached from the top-right avatar/name button on /trainer/studies. The menu opens, shows the signed-in name + email, role chips (Trainer / Student / Own trainer), an "Open settings →" link, a "Quick roles" button, and a "Sign out" button. Console + page errors were both empty at desktop 1280×900 and mobile 375×812; no horizontal overflow at mobile (scrollWidth 360 ≤ 375); text contrast on email / settings link / sign-out is ~7.09 against the menu background — well above WCAG-AA.

Two a11y bugs surfaced on the live `role="menu"` popup that violate the WAI-ARIA menu button pattern:

1. **Escape did not close the menu** (`CLOSES ON ESC: false`). The menu only closed on outside mousedown. Keyboard users who opened the menu had no way to dismiss it without tabbing/clicking.
2. **The trigger had no `aria-controls`** (`ariaControls: null`) and no accessible name beyond the user's display name; the popup `<div role="menu">` had no `id` or `aria-label` either, so assistive tech could not associate the trigger with the menu.

Fix in `src/auth/UserMenu.tsx`: added a `keydown` listener while the menu is open that closes it on `Escape` (also exits the inline RolesEditor sub-state); gave the trigger `aria-controls="user-menu"` and an `aria-label="Open profile menu"`; gave the popup an `id="user-menu"` and `aria-label="Profile menu"`. No layout, copy or visual changes.
