# Audit run — 2026-05-24T10:09:41Z
**Mode:** TEST_ONE
**Subject:** trainer-studies-list
**Result:** IMPROVED

Picked `trainer-studies-list` because it was the only feature with a null `lastTestedAt`. The spec authenticates as the bot, navigates to `/trainer/studies`, asserts the heading, the three stat tiles, and the three section headings, then runs an a11y/responsive sweep on the filter row, view-toggle, and search input.

What I saw on prod (https://praxis.tsrun.dev/trainer/studies): the page renders cleanly and there are no console or page errors. But three UI-quality issues showed up. The search input has no accessible name — only `placeholder="Find study by name or ECO…"`, with `aria-label=null` and no associated `<label>` — so screen readers do not announce the field's purpose. The grid/list view-toggle buttons are icon-only with `textContent=""`, `aria-label=null`, `title=null`, so they have no accessible name at all. Both Segmented containers (the filter row and the view-mode toggle) render with `role=null` and no `aria-label`, so the related toggle buttons have no group semantics. At 375×812, the search input collapses to roughly 30px wide because its container uses `flex: 1` (shrinks to zero) while the four-button filter Segmented eats the rest of the row, leaving a visually-present but effectively unusable search field on phones.

Decision: IMPROVE. Smallest change that addresses all four points — add `aria-label="Search studies"` on the input, give the search container a `flex: '1 1 220px', minWidth: 200` so it wraps to its own line on narrow viewports, pass `ariaLabel` to both Segmented groups (filter + view) plus per-option `ariaLabel` for the icon-only Grid/List buttons, and extend `SegmentedOption<T>` with an optional `ariaLabel` field that the underlying button consumes via `aria-label`. Typecheck, unit tests (37/37), and `npm run build` all passed locally.
