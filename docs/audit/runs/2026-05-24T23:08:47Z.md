## UI observations

Picked the highest-impact three for this rotation.

**1. `outline: 'none'` hardcoded on every input.** The shared `inputStyle` literal at the bottom of `src/trainer/ImportGamesPage.tsx` set `outline: 'none'`, which (as an inline style) overrides the global `:focus-visible { outline: 2px solid var(--accent) }` rule from `src/index.css`. Spec dump on the Chess.com username input after `.focus()`:

```
CC INPUT :focus styles: { outlineStyle: 'none', outlineWidth: '3px', outlineColor: 'rgb(236, 236, 239)', boxShadow: 'none', borderColor: 'rgba(255, 255, 255, 0.04)' }
ECO INPUT :focus styles: { outlineStyle: 'none', ... }  // same
```

Affects all six inputs on the page (Chess.com / Lichess username, Database Player, Year from, Year to, ECO, Min Elo). Sighted keyboard users get no focus indicator at all on any of these — that's a meaningful regression versus other forms in the app where the amber outline is clearly visible (e.g. trainer-students-list search input correctly shows `outlineStyle: solid, outlineWidth: 2px, outlineColor: rgb(251, 191, 36)`).

**2. No programmatic `<label>` association on any of those inputs.**

```
CC USERNAME INPUT a11y: { id: null, ariaLabel: null, ariaLabelledby: null, labelsFor: 0, wrappingLabel: null, placeholder: 'e.g. Hikaru', type: 'text' }
PLAYER INPUT  a11y: { id: null, ariaLabel: null, labelsFor: 0, wrappingLabel: null, placeholder: 'e.g. Carlsen' }
ECO INPUT     a11y: { id: null, ariaLabel: null, labelsFor: 0, wrappingLabel: null }
MIN ELO INPUT a11y: { id: null, ariaLabel: null, labelsFor: 0, wrappingLabel: null, type: 'number' }
YEAR FROM     a11y: { id: null, ariaLabel: null, labelsFor: 0, wrappingLabel: null, type: 'number' }
```

In each case the visible `<label className="meta">Chess.com username</label>` is rendered as a sibling div above the input, not a wrapper, and carries no `htmlFor`. Screen readers announce all six inputs without their purpose.

**3. The source-picker Segmented (Chess.com / Lichess / Database) and the position-mode Segmented (Free editor / Play moves) are unnamed containers.**

```
SOURCE SEGMENTED container:    { role: null, ariaLabel: null, ariaLabelledby: null }
POSITION MODE SEGMENTED:       { role: null, ariaLabel: null }
```

The `Segmented` atom already wires `role="group"` + `aria-label` to the container when `ariaLabel` is passed (verified `src/components/ui/atoms.tsx`), and per-button `aria-pressed` is already set — the call-sites just don't supply `ariaLabel`. Same pattern PR #75 fixed for `StudiesPage`.

## Observed but not fixed in this PR

- **Mobile horizontal overflow.** `MOBILE OVERFLOW: { scroll: 478, client: 375 }` — the page horizontally scrolls by ~100px on a 375-wide viewport. Probe traced it to the position-filter `Card`: the `PlayFromStartBoard` flex row (board sized `width: 100%, maxWidth: 360` next to a `BoardToolbar` sibling) doesn't shrink under the board's intrinsic 360px aspect-ratio width, pushing the card to 466 wide. Fix touches `src/components/board/PlayFromStartBoard.tsx` which is also consumed by `src/trainer/ImportGamesPage.tsx`'s scoped path. Out of scope for this rotation — leaving for a future single-purpose fix so the diff stays narrow.
- **Color / Result / Time control checkbox-radio groups** had orphan `<label>` elements acting as group titles; converted to `role="group"` + `aria-label` semantics in passing since they're trivially adjacent to the labelled inputs above.
- **Filters `<details open>` summary** has no `aria-controls` — acceptable native `<details>` semantics, not changed.
