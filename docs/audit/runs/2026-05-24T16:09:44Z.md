The diagnostic dump turned up several UI/a11y issues, but all the major ones are already being addressed by in-flight PRs:

- `MODE TABS: [{ text: "Sign in", ariaPressed: null, ... }, { text: "Create account", ariaPressed: null, ... }]` — active mode is communicated only via background lightness, not via the a11y tree. Already addressed by PR #78 (adds `role="group"` + `aria-pressed` on each tab).
- `ERROR REGION: { text: "missing fields", color: "rgb(248, 113, 113)", role: null, ariaLive: null, fontSize: "12px" }` — visible "missing fields" danger text appears on empty-submit but has no `role="alert"` / `aria-live`, so screen readers don't announce it. Already addressed by PR #78 (switches the error region to always-rendered with `role="alert"` + `aria-live="polite"`).
- `MOBILE OVERFLOW: { scroll: 382, client: 375 }` — the page horizontally overflows by 7 pixels at 375 wide (hero radial-gradient glow's `-10%` inset escapes the body). Already addressed by PR #78 (adds `body { overflow-x: clip; }`).

One UI issue is UNIQUE to this flow and not covered by PR #78:

- `FOCUS after empty-submit error: { tag: "BODY", isBody: true }` and `SUBMIT after error: { hasFocus: false }` — clicking the disabled-on-submit button causes the browser to remove focus from the button when `disabled` flips true (during the in-flight signin attempt), then when `setBusy(false)` re-enables the button focus stays on `document.body`. A sighted keyboard user loses their place; pressing Tab next lands them somewhere unrelated to the form. The fix is straightforward (refocus the submit button or first invalid input via a `useEffect` keyed on `err`, or replace `disabled={busy}` with `aria-busy={busy}` + a re-entrancy guard), but PR #78 modifies the same file extensively (adds `useId` calls at the top of the component, restructures the JSX, changes the error region to always-rendered with `aria-describedby` on the submit). Opening a second improvement PR against the same file would create merge conflicts with PR #78 just as it is queued to land.

Also noted but not blocking: the submit hover state is functional but subtle (background fades from `rgb(251, 191, 36)` to `rgb(245, 182, 17)` with a soft 8px amber shadow — a ~3.5% lightness shift); the empty-submit error message ("missing fields") is generic and doesn't identify which field is missing; inputs gain no `aria-invalid` after a validation error.
