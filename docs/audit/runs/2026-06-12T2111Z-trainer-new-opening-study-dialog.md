Exercised the New opening study dialog from `/trainer/studies` at 1280×800 and 375×812. The base a11y is solid: dialog has `role="dialog"`, `aria-modal="true"`, `aria-labelledby` pointing at the "New opening study" h2; Esc + close button + outside-click all dismiss; the Study name input wraps in a `<label>` (implicit association) and shows an amber focus ring (2px solid + 3px soft glow); the side radiogroup is `role="radiogroup"` with a labelled `aria-labelledby`; both `.role-pick` tiles match `:has(input:focus-visible)` so the visible card outlines on keyboard focus; the disabled Create button blocks empty-name submit; mobile side-option tiles are 301×70 (well above the 44px tap floor).

One real defect on **mobile**: the sticky footer (Cancel / Create study) renders **under** the chessboard pieces. `tests/audit/screenshots/trainer-new-opening-study-dialog-mobile.png` shows the bottom rank of the starting position painted over the buttons. Root cause: chessground's CSS sets `.cg-wrap piece { z-index: 2 }` on every piece (and `z-index: 11 !important` when one is being dragged), but the `.cg-wrap` element is `position: relative` with no `z-index` — so it does **not** create a stacking context. The pieces' z-index escapes into the parent (the dialog's `<form>`) stacking context, where the sticky footer has `z-index: auto`. Stack order in the form context: pieces (z=2) > footer (z=auto). Visually the pieces paint on top of the footer; functionally the click still lands on the button (pieces have `pointer-events: none`), so it reads as a purely visual regression — but it looks broken.

Fix in `src/trainer/NewOpeningStudyDialog.tsx` (+8 LOC, no logic change): give `PrefixBoard`'s outer `<div>` (the chessground wrapper) `isolation: 'isolate'`. That creates a stacking context that contains every chessground z-index (including the dragging piece at 11), so the sticky footer below it paints above by DOM order — no z-index magic numbers needed. Verified with the existing audit spec on prod (TOPMOST AT CREATE-BTN CENTER already returns the button because of `pointer-events: none`, so this is a visual-only fix; the diff change is safe to deploy regardless). Typecheck passes, all 48 unit tests pass. PR opened.
