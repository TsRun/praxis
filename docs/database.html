Database — conceptual model

Two halves living in the same Postgres database: the ChessCoach app tables (users, sessions, mentor links, studies, assignments, attempts) on top, and the OpeningTree library tables (player / game / game_move / move_stats) underneath. The library is read-only from the app's point of view — it's the explorer trainers browse when building opening studies.

ChessCoach (app tables)

One unified app_user table. Every account has a multi-select roles array (trainer, student, self); sign-in is single-form, role choice is on signup. A user with the trainer role can run a roster of other users (linked via mentor); a user with the self role is their own coach (studies they author are auto-assigned to themselves); a user with the student role can be added to a trainer's roster and receive assignments.

Queries the schema is built for

fast path GET /api/explorer?fen=… (no filter) ~150 ms @ 2.35M games 
SELECT san, uci, child_fen,
       games, white_wins, draws, black_wins,
       rating_sum, rating_n
  FROM move_stats
 WHERE parent_fen = $1
 ORDER BY games DESC;
filtered path GET /api/explorer?fen=…&player_id=…&color=white ~sub-sec post-denorm 
SELECT san, uci, child_fen,
       COUNT(*) AS games,
       SUM((result = 'W')::int) AS white_wins,
       SUM((result = 'D')::int) AS draws,
       SUM((result = 'B')::int) AS black_wins,
       AVG(NULLIF(mover_elo, 0))::int AS avg_elo
  FROM game_move
 WHERE parent_fen = $1
   AND white_fide_id = $2          -- partial index lookup
 GROUP BY san, uci, child_fen
 ORDER BY COUNT(*) DESC;
autocomplete GET /api/players?q=carl ~10-30 ms 
SELECT p.fide_id, p.name, p.country, p.title, p.rating,
       similarity(p.name, $1) AS sim,
       (SELECT COUNT(*) FROM game g
          WHERE g.white_fide_id = p.fide_id OR g.black_fide_id = p.fide_id) AS games
  FROM player p
 WHERE p.name ILIKE $1 || '%' OR p.name % $1   -- prefix + trgm
 ORDER BY games DESC, sim DESC, p.name ASC
 LIMIT 12;

Lifecycle

Ingest. scripts/ingest.ts streams a PGN, derives a stable fingerprint per game, skips fingerprints already in game, inserts the new row, and writes one game_move row per ply (with the four denormalized columns filled inline). PGN-only FIDE IDs trigger lazy player stubs marked origin='pgn'; the next npm run load-fide upgrades them with the real FIDE record.

Rebuild. After a batch, scripts/rebuild-stats.ts rebuilds move_stats via one big GROUP BY into an UNLOGGED staging table, then atomically renames it in. The explorer stays live throughout — readers either see the previous summary or the new one, never a half-built state.

Refresh. scripts/load-fide.ts pulls players_list_xml.zip from FIDE, stream-parses it with sax-js, and upserts 1.84M rows. origin='pgn' rows are promoted to origin='fide' on conflict.

Diagram is a custom Crow's-foot / Merise-flavored representation. The authoritative DDL lives in server/schema.sql; the autoritative ingest in scripts/ingest.ts; the derived-table rebuild in scripts/rebuild-stats.ts.
